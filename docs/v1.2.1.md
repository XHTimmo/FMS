# Studio Financial System v1.2.1 系统架构与业务逻辑全景文档

## 1. 系统架构概述

Studio Financial System 是一个基于 **Electron** 构建的现代化跨平台桌面应用，专为工作室等小型组织的财务与会员管理设计。系统采用前后端分离但同构在本地的架构：

- **前端层 (Renderer)**：使用 **Vue 3 (Composition API)** + **Vite** 构建，UI 框架采用 **Element Plus**，数据可视化采用 **ECharts**。负责页面渲染、用户交互、数据验证和图表展示。
- **后端层 (Main)**：基于 **Node.js** 运行在 Electron 的主进程中。负责与操作系统底层的交互，包括本地文件系统操作、数据导出生成（PDF/Excel）、自动更新（electron-updater）。
- **数据层 (Database)**：采用 **SQLite3**（通过 `better-sqlite3` 驱动）进行本地化存储。数据完全保存在用户本地，确保隐私与安全。
- **通信层 (IPC)**：通过 `contextBridge` 和 `ipcMain`/`ipcRenderer` 构建类型安全的通信桥梁，前端通过 `window.electronAPI.invoke` 调用后端服务。

## 2. 核心模块说明

系统主要划分为四大核心模块：

### 2.1 数据看板 (Dashboard)
- **功能**：提供财务总览，包括总资产、总收支、净收入计算，支持按月、季度、年及自定义日期范围进行多维度的收支趋势分析和项目构成统计。
- **特色**：支持将报表数据无损导出为 PDF、Excel（包含图表）和自定义分辨率的高清 PNG。

### 2.2 收支记账 (Transactions)
- **功能**：管理日常收入与支出记录。支持关联收支分类、关联会员。
- **变更提示**：在最新的逻辑中，**取消了用户侧的“账户”设定**，界面上不再要求选择或展示资金账户，系统底层自动使用默认账户兜底，极大简化了用户的记账流程。

### 2.3 会员管理 (Members)
- **功能**：管理工作室的会员信息及缴费情况。
- **特色**：会员缴费与财务系统深度打通。会员缴纳会费时，会自动在“收支记账”中生成对应的收入账单；撤销缴费时，对应的财务账单会自动回滚（软删除）并扣减账户余额。

### 2.4 系统设置 (Settings)
- **功能**：管理系统级配置。支持自定义数据存储路径、手动数据备份、查看包含详细操作人和 IP 的系统审计日志，以及执行应用的在线热更新（OTA）。

## 3. 业务流程与数据流转逻辑

### 3.1 会员缴费与财务联动流程
1. **用户操作**：在“会员管理”界面为指定会员添加一条缴费记录。
2. **前端流转**：校验表单后，调用 IPC 接口 `member:fee:pay`。
3. **后端事务 (Transaction)**：
   - 开启 SQLite 数据库事务 (`db.transaction`)。
   - 插入一条 `membership_fees` 记录。
   - 系统自动寻找默认分类（如“会费收入”）和默认资金账户（兜底 ID 为 1）。
   - 在 `transactions` 表中插入一笔对应的收入账单，并绑定刚刚创建的缴费记录。
   - 更新对应账户的余额 (`balance = balance + amount`)。
   - 记录审计日志 (`audit_logs`)。
   - 事务提交。
4. **异常处理**：若任何一步失败，整个事务回滚，确保数据一致性。

### 3.2 报表数据导出流程
1. **指令发起**：用户在 Dashboard 点击“导出 PDF/Excel”。
2. **图表捕获**：前端利用 `html2canvas` 或 ECharts 实例，将当前的动态图表转换为 Base64 图片数据。
3. **IPC 传输**：前端将表格明细数据和图表 Base64 一并发送至后端。
4. **后端渲染**：
   - **Excel**：使用 `exceljs` 动态生成工作簿，插入表格数据并嵌入 Base64 图片，保存为 `.xlsx` 文件。
   - **PDF**：创建一个隐藏的 Electron `BrowserWindow`，加载注入了数据的 HTML 模板，调用 `webContents.printToPDF()` 渲染出无损 PDF 并保存。
5. **结果返回**：后端将生成的文件路径返回给前端，前端弹出成功提示并可直接打开该文件。

## 4. 关键算法实现

- **默认账户兜底算法**：为响应“取消账户设定”的需求，在处理账单写入时，若前端未传 `account_id`，后端执行 `SELECT id FROM accounts ORDER BY id LIMIT 1` 获取默认账户。这在不破坏原有数据库外键约束和数据统计的前提下，实现了 UI 层面的简化。
- **分页与全量数据平滑切换**：看板展示时使用 `LIMIT` 与 `OFFSET` 实现分页以提升渲染性能；在导出时，系统自动识别并执行无限制的查询（或指定日期区间内的全量查询），确保导出的报表不会遗漏数据。
- **基于事务的级联回滚**：在撤销会员缴费时，通过 `fee.id` 反向查找对应的 `transaction`，将其 `status` 置为 `deleted`，并根据该交易的金额与类型反向修正账户余额，该过程全部封装在一个数据库事务中完成。

## 5. 接口定义 (IPC 契约)

系统核心 IPC 接口列表如下：

| 通道名称 (Channel) | 功能描述 | 请求参数 | 返回类型 |
| :--- | :--- | :--- | :--- |
| `transaction:list` | 获取账单列表 | `{ period?, year?, month?... }` | `Transaction[]` |
| `transaction:create` | 创建新账单 | `TransactionPayload` | `{ id: number }` |
| `member:fee:pay` | 缴纳会费 | `{ member_id, amount, ... }` | `void` |
| `dashboard:stats` | 获取看板统计 | `{ periodType, startDate, endDate }` | `DashboardStats` |
| `export:pdf` | 导出PDF报表 | `{ html, path? }` | `{ success: boolean, path: string }` |
| `system:backup:run`| 运行系统备份 | `void` | `{ success: boolean, backupPath: string }` |
| `system:audit:list`| 获取审计日志 | `{ limit?: number }` | `AuditLog[]` |

## 6. 异常处理机制

- **主进程异常拦截**：所有 `ipcMain.handle` 均包裹了 `try...catch (error: unknown)`，利用类型收窄机制提取错误信息（`error instanceof Error ? error.message : String(error)`），确保不会因为未知的抛出物导致主进程崩溃。
- **原子性保障**：凡是涉及多表联动的写入操作（如缴费、退费、修改账户与流水），全部在 `db.transaction()` 中执行。遇到错误时自动触发回滚。
- **UI 友好提示**：前端拦截所有的 IPC 调用异常，通过 `Element Plus` 的 `ElMessage.error()` 抛出用户可读的中文错误提示（如“补偿同步失败：缺少分类或账户”），避免生硬的代码级报错暴露给终端用户。
